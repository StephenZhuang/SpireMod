该模组（SpireMod）在错误处理上采用了一种**轻量级、非中断式**的策略，主要依赖于**布尔返回值**进行逻辑控制，并辅以**控制台打印堆栈**作为最后的兜底手段。由于是《杀戮尖塔》的客户端模组，其错误处理旨在避免游戏崩溃，而非提供复杂的用户交互或事务回滚。

### 1. 核心策略：布尔返回值与前置检查
业务逻辑层（如 `LoanState`）不抛出异常，而是通过返回 `boolean` 来指示操作成功与否。调用方（如 `ShopLoanPatch`）负责在执行前进行状态校验（Guard Clauses）。
- **模式**：`if (!canBorrow()) return;`
- **优势**：避免了异常捕获的性能开销，符合游戏高频循环（Update/Render）的性能要求。
- **实现**：`LoanState.borrow()` 和 `repay()` 均返回 `false` 表示失败，调用方根据返回值决定播放“购买成功”还是“无法购买”音效。

### 2. 异常捕获：静默失败与堆栈打印
在涉及 I/O 操作（存档读写）或反射调用时，模组采用 `try-catch` 块捕获所有异常。
- **处理方式**：捕获后仅调用 `e.printStackTrace()`，不进行上层抛出或复杂的重试逻辑。
- **场景**：
  - `LoanSavePatch`：在读写 `loanstate.dat` 文件时，若发生 `IOException` 或 `NumberFormatException`，打印错误并重置状态或忽略写入。
  - `GoldPatch` / `LoanSavePatch`：在通过反射访问游戏私有方法 `SaveHelper.getSaveDir()` 失败时，打印错误并返回 `null`。
- **约束**：这种“静默失败”策略确保了即使模组内部出错，也不会导致主游戏进程崩溃。

### 3. 构建期错误：显式中断
与运行时的宽容不同，构建脚本（`build.gradle`）对环境缺失采取**严格中断**策略。
- **实现**：若找不到《杀戮尖塔》或 ModTheSpire 的 JAR 包，直接抛出 `GradleException`。
- **目的**：确保开发环境的完整性，防止生成无效的模组文件。

### 4. 开发者规范
- **禁止抛出运行时异常**：在游戏逻辑补丁（Patches）中，严禁抛出未受检异常，以免触发游戏的异常捕获机制导致弹窗或崩溃。
- **I/O 操作必须包裹 Try-Catch**：所有文件读写必须自行处理异常，并通过 `printStackTrace` 记录日志以便调试。
- **状态校验前置**：在执行增益、借贷等状态变更前，必须先调用 `LoanState` 提供的 `canXxx()` 方法进行预检。
